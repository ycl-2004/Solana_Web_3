============

Workflow
--------

完整顺序：
1. `npx ts-node src/index.ts`
   检查 RPC、钱包、余额

2. `npx ts-node src/uploadCollectionMetadata.ts`
   上传 collection 的图片和 metadata

3. `npx ts-node src/createLegacyCollection.ts`
   创建 Collection NFT

4. `npx ts-node src/uploadLegacyNftMetadata.ts`
   上传普通 NFT 的图片和 metadata

5. `npx ts-node src/mintLegacyNft.ts`
   铸造普通 NFT，并 verify 到 collection

一句话理解：
- 前 3 步是在创建“系列”
- 后 2 步是在创建“系列里的成员 NFT”

4. Collection 和 Verify
-----------------------

关键理解：
- Collection = 系列的官方主 NFT
- Verify = 官方确认某张普通 NFT 真的属于这个系列

注意：
- 创建 Collection 不等于已经 verify
- Collection 本身不需要再 verify 一次
- 以后是普通 NFT 先挂上 collection，再由 authority 去 verify

7. 这一步和上一步的区别
-----------------------

上一步：
- 创建的是 Collection NFT
- 它是整个系列的根
- 它代表“future 这个系列本身”

这一步：
- 创建的是普通 NFT
- 它是系列里的某一个成员
- 它需要被挂到 collection 上，并经过 verify

一句话理解：
- 上一步是在创建“系列”
- 这一步是在创建“系列里的作品”

9. Anchor Custom Minter 总结
----------------------------

目标：
- 不再用本地脚本手动一步一步 mint
- 改成写一个自己的 Anchor 程序
- 让用户调用程序后，程序自动执行 mint 规则

这条线现在完成了什么：
- 建立独立 Anchor 项目：`NFT/anchor-minter`
- 初始化链上总配置 `MintConfig`
- 实现免费 mint
- 实现每钱包最多 mint 1 张
- 实现销毁后可再 mint
- 接上 Metaplex metadata
- 接上 master edition
- 接上 collection authority delegate
- 接上自动 verify 进现有 collection

一句话理解：
- Legacy 那一段是“你手动操作”
- Anchor 这一段是“把规则写进链上程序，让程序自动操作”


10. Anchor Workflow
-------------------

本地开发顺序：
1. `cd NFT/anchor-minter`
2. `anchor build`
3. 启动本地 validator
4. `solana program deploy target/deploy/anchor_minter.so --program-id target/deploy/anchor_minter-keypair.json --url http://127.0.0.1:<PORT>`
5. `ANCHOR_PROVIDER_URL=http://127.0.0.1:<PORT> ANCHOR_WALLET=~/.config/solana/id.json npx ts-mocha -p ./tsconfig.json -t 1000000 "tests/**/*.ts"`

devnet 实际使用顺序：
1. `cd NFT/anchor-minter`
2. `anchor build`
3. `npx ts-node scripts/approveCollectionAuthority.ts`
4. `solana program deploy target/deploy/anchor_minter.so --program-id target/deploy/anchor_minter-keypair.json --url devnet`
5. `ANCHOR_PROVIDER_URL=https://api.devnet.solana.com ANCHOR_WALLET=~/.config/solana/id.json npx ts-node scripts/initializeConfig.ts`
6. `ANCHOR_PROVIDER_URL=https://api.devnet.solana.com ANCHOR_WALLET=~/.config/solana/id.json npx ts-node scripts/mintNftWithMetadata.ts`

说明：
- 第 3 步 delegate 只需要做一次
- 第 5 步 initialize config 只需要做一次
- 第 6 步是以后真正给用户 mint 的逻辑入口

12. Anchor 权限结构
-------------------

现在的权限关系：
- 主钱包仍然是 collection 真正的 owner / update authority
- 程序的 `config PDA` 是 collection delegate
- 程序可以自动替你 verify 新 NFT
- 用户不需要每次来找你手动盖章

一句话理解：
- 钱包还是老板
- 程序是你授权出去的管理员


13. 其他人 mint 时需要上传什么
------------------------------

如果用你现在这套 `mintNftWithMetadata`：
- 他们不需要自己上传图片
- 他们也不需要自己上传 JSON
- 现在脚本里已经写死了一份 metadata URI，所以 mint 出来的内容会是同一套模板

如果你以后想让“每个人 mint 出不同内容”，才需要额外设计：
- 用户自己先上传图片和 metadata
或
- 由你的后端 / 前端 帮他们上传，再把 `uri` 传给程序

你当前这版更像：
- 免费领取同一个系列模板下的一张 NFT

不是：
- 用户自己上传素材，生成完全自定义 NFT
